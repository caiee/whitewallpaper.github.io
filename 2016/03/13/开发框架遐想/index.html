开发框架啰嗦

发表于 2016-03-13   |   分类于 Framework

前言

这篇博客用来记录使用开发框架时的一些tips，
是因为在宅米工作了半年时间，感觉到框架方面很多都用的不是很好(平时开发)，
也可能是工程师都比较年轻，资历尚浅。
同时我自己也想多了解了解框架，并记录一下。

主要围绕以下几个框架：
1、 Spring MVC
2、 Mybatis
3、 Spring
4、 Junit

正文

1、 Spring MVC

Spring MVC是Spring提供的WEB框架，与之相比较的是Struts等，还有Jersey。
但Struts自从出了那个安全漏洞之后，用它的公司就越来越少了。
Jersey也是一个蛮不错的框架，在我哥公司的项目中看到过学习过。
不过Jerse和Spring MVC大体上给我的感觉都差不多。
下面仅仅是讲讲个人心得.

这是宅米用户APP关于用户注册的后台API，从这段代码可以发现很多问题。

1. 方法的入参是一个 Map类型的Key，Value均为Object的值。

1) 意味着这个方法可以接收无限多的参数，而框架面对无限多的参数时，也不需要去考虑什么，直接转为Object放进Map里就好了。
2) 同事想来维护这段代码时，单看这个入参，无法快速的得到从前端接收了哪些参数。还得去看代码，看从这个Map拿出了哪些参数。
3) 由于是一个Value为Object的Map，所以在方法的代码中会出现很多(Something)map.get(“something”)这样的nosiy code，导致代码的不清晰，维护成本高等等。
应对：
对于这个方法所接受参数，创建一个pojo类，将每一个参数都列为pojo类中的字段，方法入参时用具体的pojo类来接受(框架会帮我们自动转)。

3. 参数非空判断与长度等校验逻辑放在了方法中，这也是极不合理的，造成方法的nosiy code极多，方法极难理解。

应对：
可以借助hibernate-validate完成。前提是接受的参数是一个pojo类，在这个类的每一个需要校验的字段上添加上我们需要的逻辑即可。

4. 一个方法中有多处出口，这是反模式的。

在学术上来说，一个方法最好的情况是一个出口。
这个情况仁者见仁智者见智吧。

5. 方法的返回值是一个ReturnValue，so what is returnvalue???

被一个交大的哥们吐槽之后，他用一己之力拯救了我们。。。
实现了Spring MVC提供的ResponseBodyAdvice，在输出前再做一层包装。
这样的话我们的方法就可以随意返回想要的东西了，更加清晰。

By the way

Spring 4也支持了 Java8 中的Optional，这意味着我们可以在方法的入参中使用Optional类。

这意味着我们在方法的代码中可以潇洒的使用，
accountId.ifPresent(accountId -> doSomething);
热爱函数式编程猿的最爱！
遗憾是的Mybatis并没有提供这样的支持，
我们还是需要在Mybatis查出来的数据上作 if (result == null) 类似的判断，
这种nosiy code，真的很让猿心伤。。希望Mybatis能够更新并支持它吧！

2. Mybatis

Mybatis和Hibernate一样，都是数据库框架，
而前者更是称为只需要懂Java和Sql就可以用好的一个框架，维护也更加方便；
后者相对门槛高得多，对象的级联关系等等，一不小心就是一个递归删除。

然而我发现公司的项目在使用Mybatis的时候，很多框架提供的特性和精华都没有用到，
比如dao层写了实现类，这其实是多余的，因为mybatis提供动态代理将接口和xml关联。
造成出现原本很多本不应该出现的代码，nosiy code..

4 mybatis的两种用法
它支持在mapper的接口上直接写注解@insert @update @delete @select等注解，
不过那样写并不简单明了，且resultMap之类的通过注解也不能重用，

我个人更倾向于把sql语句全部写在mapper.xml中，更加清晰，也易于管理，维护等。

3. Spring

一个非常强大的一站式框架，什么都可以一起配合着用。
用法到都差不多，注册到Spring容器中，由Spring容器帮我们将具体的Bean注入到具体的类的某个字段中。
只是在项目中关于Spring的配置文件做的不是很好，
它的配置文件应该以applicationContext为开头，后面跟上 - ，再加上具体配置的相关内容，
比如dataSource的那就是applicationContext-dataSource.xml，
redis的则是applicationContext-redis.xml，
在引用的时候用通配符 applicationContext-*引用，
或者是在主的applicationContext中 其他的配置文件。
这样做可以让配置文件更加易懂，一定要避免所有的配置都堆在一个文件里。

4. Junit

Junit也是一个非常强大的框架，他可以帮助我们快速的运行我们的代码。
@Test，将这个注解放在需要运行的方法上，即可快速运行这段代码。

在项目中的话主要起着一个单元测试的作用，也就是和项目整合在一起，快速的将spring配置文件加载进来，
并运行项目代码。
在整合的过程中有两个主要的注解

在日常的单元测试中，比较推荐制定一个BaseApplicationContext，用来加载Spring的配置文件

那在单元测试中最重要的事情则是代码的覆盖率，
一般会通过 Assert 类来断言我们输入的参数与应该输出的参数是否相等。
而那些应该抛出异常的边界情况，则使用的是@Test(expected = RuntimeException.class)来判断。
