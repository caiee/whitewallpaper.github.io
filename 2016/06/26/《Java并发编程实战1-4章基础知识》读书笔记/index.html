简介

并发简史

为什么需要并发

充分 发挥 多处理器系统的强大计算能力。

  • 资源利用率
    • 同时运行多个程序， 提高资源利用率
  • 公平性
    • 粗粒度时间分片， 让用户和程序共享计算机资源，而非一个程序从头到尾运行，然后再启动下一个程序
  • 便利性
    • 为计算多个任务编写多个程序， 它们之间必要时互相通信

高效：在串行性和异步性之间找到合理的平衡。

线程的优势

一

  • 降低程序开发和维护成本
  • 提升复杂程序性能
  • 将大部分异步工作流转换成串行工作流
  • 降低代码复杂度，更易编写，阅读，维护

发挥多处理器的强大能力

多处理器的原因： 通过提高始终频率来提升性能变得越来越困难，厂商转而在单个芯片上放置多个处理器核。

日益普及的多处理器系统， 价格也在不断的降低。

基本的调度单位： 线程 ，
一个线程最多同时只能在一个处理器上运行，多处理器时则会造成浪费。
如果设计正确，多线程程序可以通过提高处理器资源的利用率来提升系统吞吐率

单线程程序，等待I/O时，处理器将处于空闲状态。
多线程程序，等待时另一个程序可以继续运行(烧水时看报，而不是烧开了再看报)。

建模简单性

通过使用 线程，可以将复杂并且异步的工作流进一步分解为一组简单并且同步的工作流，每个工作流在一个单独的线程中运行，
并在特定的同步位置交互。

异步事件的简化处理

线程带来的风险

双刃剑

安全性问题

存在指令从排序的问题.

活跃性问题

安全性 含义 是 永远不发生糟糕的事情
活跃性 含义 是 某件正确的事情最终会发生

性能问题

多线程程序，当线程调度器临时挂起活跃线程并转而运行另一个线程时，就会频繁地出现上下文切换操作 (Context Switch)，
这种操作将带来极大的开销： 保存和恢复执行上下文，丢失局部性，并且CPU时间将更多地花在线程调度而不是线程运行商。
当线程共享数据时，必须使用同步机制，而这些机制往往会抑制某些编译器优化，使内存缓存区中的数据无效，
以及增加共享内存总线的同步流量。

线程无处不在

框架通过在框架线程中调用应用程序代码将并发性引入到程序中。在代码中将不可避免地访问应用程序状态，
因此所有访问这些状态的代码路径都必须是线程安全的。

线程安全性

“共享“ 意味着 变量可以由多个线程同时访问
“可变“ 意味着 变量的值在其生命周期内可以发生变化

一个对象是否需要是 线程安全 的， 取决于是否被多个线程访问。

同步关键字 synchronized，
术语：

  • volatile 类型变量
  • 显示锁(Explicit Lock)
  • 原子变量

程序没有使用合适的同步导致问题时，可以采取的解决办法：

  • 不在线程之间共享该状态变量
  • 将状态变量修改为不可变的变量
  • 在访问状态变量时使用同步

什么是线程安全性

当多个线程访问某个类时，这个类始终都能表现出正确的行为，那么就称这个类是线程安全的。

无状态对象一定是线程安全的，计算过程中的零食状态仅存在于线程栈上的局部变量中，由正在执行的线程访问。

原子性

原子操作：
对于访问同一个状态的所有操作(包括该操作本身来说)， 这个操作是一个以原子方式执行的操作。

竞态条件

当某个计算的正确性取决于多个线程的交替执行时序时，就会发生竞态条件。

最常见的竞态条件类型： 先检查后执行

复合操作

要避免 竞态条件 问题，就必须在某个线程修改该变量时，通过某种方式防止其他线程使用这个变量，
从而确保其他线程只能在修改操作完成之前或之后读取和修改状态，而不是在修改状态的过程中。

java.util.concurrent.atomic -> 原子变量类， 用于实现在数组和对象引用上的原子状态转换。
AtomicLong

加锁机制

AtomicReferenc 替代对象引用的线程安全类

内置锁

互斥体 (互斥锁)， 最多只有一个线程可以持有锁。

同步代码块

  • 锁的对象引用
  • 锁保护的代码块

每一个对象都可以是实现同步的锁，称为 内置锁(Intrinsic Lock) 或 监视器锁 (Monitor Lock)。
线程进入代码块获取锁，退出代码块释放锁。

重入

某个线程试图获得一个已经由它自己持有的锁。
重入 提升加锁行为的封装性，简化面向对象并发代码的开发。

用锁来保护状态

每个共享和可变的变量都只能由一个锁来保护。
对于每个包含多个变量的不变性条件，其中涉及的所有变量都需要由同一个锁来保护。

活跃性与性能

不良并发( Poor Concurrency ) ：
可同时调用的数量，不仅受到可用处理资源的限制，还受到应用程序本身结构的限制。

缩小同步代码块的代码数量到最少

明确要同步的是哪一块东西，尽可能缩小代码块，保持简单且性能高效。

当执行时间较长的计算或可能无法快速完成的操作 (I/O)， 一定不要持有锁。

如何安全地共享和发布对象

同步 ： 内存可见性( Memory Visibility)

可见性

重排序(Reordering)

失效数据

get 和 set 都没有同步： 某个线程调用set， 另外一个调用get的线程可能会看到更新后的值，也有可能看不到。

两个方法都同步，保证线程安全以及正确性。

非原子的64位操作

没有同步的情况读取变量发生的情况，称为 最低安全性 (out-of-thin-airsafety)。

Java内存模型要求， 变量的读取操作和写入操作都必须是原子操作，但对于非volatile类型的long和double变量，
JVM允许将64位的读操作或写操作分解为两个32位的操作。

加锁与可见性

加锁的含义不仅仅局限于互斥行为，还包括内存可见性。为了确保所有线程都能看到共享的变量的最新值，
所有执行读操作或者写操作的线程都必须在同一个锁上同步。

Volatile变量

Java 提供了 稍弱的同步机制 ： volatile 变量，用来确保将变量的更新操作通知到其他线程。

通常用作某个操作完成、发生中断或者状态的标志。（ volatile 的语义不足以确保递增操作(count++) 的原子性。
原子变量提供了 “ 读 - 改 - 写 “ 的原子操作，并且常常用作一种 更好 的 volatile变量。

加锁机制既可以确保可见性又可以确保原子性，而volatile变量只能确保 可见性。

使用volatile变量的条件：

  • 对变量的吸入操作不依赖变量的当前值，或者你能确保只有单个线程更新变量的值。
  • 该变量不会与其他状态变量一起纳入不变性条件中。
  • 在访问变量时不需要加锁

发布与逸出

发布 (Publish) ： 使对象能够在当前作用域之外的代码中使用。
逸出 (Escape)： 当某个不应该发布的对象被发布。

发布对象最简单的方式是将对象的引用保存到一个公有的静态变量中，以便任何类和线程都能看见对象。

使内部的可变状态逸出

安全的对象构造过程

不要在构造过程中使this引用逸出。

线程封闭

共享可变数据时，需要同步，避免使用同步则是不共享数据；
只在单线程内访问数据，则不需要同步，这种技术称为线程封闭 ( Thread Confinement)。

Ad-hoc线程封闭

指维护线程封闭性的职责完全由程序实现承担。

在volatile变量上存在一种特殊的线程封闭，如果能确保只有单个线程对共享的volatile变量执行写入操作，
那就可以安心的在这些共享的变量上执行 “ 读取 - 修改 - 写入”，
相当于将修改操作封闭在单个线程中以防止发生竞态条件，并且
volatile变量的可见性保证了其他线程能接收到最新的值。

Ad-hoc技术脆弱，尽量少用，使用更强的线程封闭技术(栈封闭， ThreadLocal类)

栈封闭

局部变量访问对象
封闭在执行线程中，其他线程无法访问这个对象。
比 Ad-hoc 线程封闭更，易于维护 也更加健壮。

ThreadLocal

使线程中的某个值与保存值的对象关联起来，
为每个使用该变量的线程都存有一份独立的副本。

通常 用于 防止对可变的单实例变量 (Singleton) 或 全局变量进行共享。

它会降低代码的 可重用性 ，并在类之间引入隐含的 耦合性 ，使用时需多加思考。

不变性

对象在被创建后的状态就不能被修改，称为不可变对象。 - 绝对的一定线程安全。

成为不可变对象的条件：

  • 创建后状态不可更改
  • 所有域都是 final 类型
  • 对象是正确创建的 (创建期间， this 引用没有 逸出)

Final域

除非需要更高的可见性，否则应将所有的域都声明为 私有域 是 一个良好的编程习惯；
除非需要某个域是可变的，否则应将其声明为 final 域 也是一个良好的编程习惯。

安全发布

任何线程都可以在不需要额外同步的情况下安全地访问不可变对象，即使在发布这些对象时没有使用同步。

常用模式

安全发布对象，要求对象引用和对象状态同时对其他线程可见，有以下几种方式：

  • 在静态初始化函数中初始化一个对象引用
  • 将对象的引用保存到volatile类型的域或者AtomicReference对象中
  • 将对象的引用保存到某个正确构造对象的final类型域中
  • 将对象的引用保存到一个由锁保护的域中

线程安全库中容器类提供了以下安全发布保证：

  • Hashtable | synchronizedMap | ConcurrentMap
  • Vector | CopyOnWriteArrayList | CopyOnWriteArraySet | synchronziedList | synchronziedSet
  • BlockingQueue | ConcurrentLinkedQueue

Future , Exchanger 也可以安全发布。

发布一个静态构造对象，最简单最安全的方式 ： 使用静态初始化器
public static Holder holder = new Holder(42);

JVM在类的初始化阶段执行， 由于JVM内部有同步机制，因此通过这种方式初始化的任何对象都可以被安全的发布。

事实不可变对象

定义： 对象从技术上来看是可变的，但其状态在发布后不会再改变

在没有额外的同步情况下，任何线程都可以安全地使用被安全发布的事实不可变对象

可变对象

可变对象，发布时需同步，访问时也需要同步来确保可见性。
安全的共享可变对象，则对象须被安全发布，并且由线程安全的类或者锁保护起来。

对象的发布需求取决于它的可变性：

  • 不可变对象可以通过任意机制来发布
  • 事实不可变对象必须通过安全方式来发布
  • 可变对象必须通过安全方式来发布，并且必须是线程安全的或者由某个锁保护起来

安全地共享对象

策略：

  • 线程封闭更
  • 只读共享
  • 线程安全共享
  • 保护对象

对象的组合

设计线程安全的类

  • 找出构成对象状态的所有变量
  • 找出约束状态变量的不变形条件
  • 建立对象状态的并发访问管理策略

同步策略 (Synchronization Policy) : 如何在不违背对象不变条件或后验条件的情况下对其状态的访问操作进行协同。

收集同步需求

如果不了解对象的不变性条件与后验条件，那么就不能确保线程安全性。 要满足在状态变量的有效值或状态转换上的各种约束条件， 就需要借助于原子性与封装性。

依赖状态的操作

基于状态的先验条件 ( Precondition )

实现某个等待先验条件为真时才执行的操作

  • 阻塞队列 Blocking Queue
  • 信号量 Semaphore

状态的所有权

垃圾回收机制使我们避免了如何处理所有权的问题。

防止多个线程在并发访问同一个对象时产生的相互干扰：

  • 线程安全的对象
  • 事实不可变对象
  • 由锁保护的对象

实例封闭

Instance Confinement ( 实例封闭机制 ）

将数据封装在对象内部，可以将数据的访问限制在对象的方法上，从而更容易确保线程在访问数据时总能持有正确的锁。
封闭机制更易于构造线程安全的类，因为当封闭类的状态时，在分析类的线程安全性时就无须检查整个程序。

Java监视器模式

通过一个私有锁来保护状态

私有锁好处：

  • 封装锁，客户端无法得到锁
  • 如果错误的获得了锁， 就会产生活跃性问题

线程安全性的委托

如果认为调用这个方法返回的数据在下文不会再有变化，那就坚决返回一个不可变对象。

独立的状态变量

将线程安全性委托给多个状态变量，只要这些变量是彼此独立的，组合而成的类并不会在其包含的多个状态变量上增加任何不变性条件。
使用 CopyOnWriteArrayList

当委托失效时

如果一个类是由多个独立且线程安全的状态变量组成，并且在所有的操作中都不包含无效状态转换，那么可以将线程安全性委托给底层的状态变量。

发布底层的状态变量

如果一个状态变量是线程安全的，并且没有任何不变性条件来约束它的值，在变量的操作上也不存在任何不允许的状态转换，那么久可以安全地发布这个变量。

在现有的线程安全类中添加功能

若没有 则 添加 (Put-If-Absent)方法

java 8 对 map 也进行了 增强 ，
增加了一些原子方法
如：

  • computeIfAbsent
  • computeIfPresent
  • putIfAbsent
  • compute

But : “扩展”类 导致 同步策略实现被分布到多个单独维护的源代码中，使得类更加脆弱。

客户端加锁机制

非线程安全的 若没有则添加

原因 ： 只有同步的假象，实际上 putIfAbsent 对于 List 的其他操作 来说 并不是 原子 的，
虽然所有的链表操作都被声明 为 synchronzied ， 但却使用了不同的锁。

客户端加锁 实现 正确的 “若没有则添加”

客户端加锁 和 扩展类加锁 共同点：

  • 将派生类的行为与基类的实现耦合在一起
  • 扩展破坏实现的封装性
  • 客户端加锁破坏同步策略的封装性

组合

组合： 为现有类 更好 地添加一个原子操作

主要是 通过 构造函数 来 初始化， 之后的所有操作通过 类名 来 使用，而不会再有客户端直接使用的情况。

将同步策略文档化

文档中说明客户代码需要了解的线程安全性保证， 以及代码维护人员需要了解的同步策略。

设计同步策略时需考虑的因素：

  • 哪些变量声明为 volatile 类型
  • 哪些变量 用 锁 保护
  • 哪些 锁 保护 哪些变量
  • 哪些 变量 必须是不可变 或 应封闭在线程中
  • 哪些 操作 必须 是原子操作

解释含糊的文档
一定一定要在文档中把细节都说清楚，而不应该让使用者去猜测 哪些是 安全 而 哪些 是不安全的.
